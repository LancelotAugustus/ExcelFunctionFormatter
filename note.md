## GitHub 版本控制常见规范

在团队协作中，遵循统一的版本控制规范能够提高开发效率、保证代码质量，并降低沟通成本。以下基于 Git 和 GitHub
平台总结的常见规范，涵盖分支管理、提交信息、代码审查、版本发布等方面。

---

### 一、提交信息规范

清晰、结构化的提交信息有助于追溯历史、自动生成 changelog。

#### 1. 格式建议

采用 [Conventional Commits](https://www.conventionalcommits.org/) 规范：

```
<类型>[可选作用域]: <描述>

[可选正文]

[可选脚注]
```

- **类型**：
    - `feat`: 新功能
    - `fix`: 修复 bug
    - `docs`: 文档变更
    - `style`: 代码格式（不影响代码含义）
    - `refactor`: 重构
    - `perf`: 性能优化
    - `test`: 增加测试
    - `chore`: 构建过程或辅助工具变动
    - `revert`: 回退
    - `others`: 其他
- **作用域**：可选，表示影响的范围（如模块名）。
- **描述**：简短总结，不超过 50 字符，首字母小写，不加句号。
- **正文**：可选，详细说明变更动机、对比等，每行不超过 72 字符。
- **脚注**：可选，通常用于关联 Issue（如 `Closes #123`）或说明不兼容变更。

#### 2. 示例

```
feat(auth): add login with GitHub OAuth

Implement OAuth flow for GitHub authentication. Users can now log in using their GitHub account.

Closes #45
```

#### 3. 其他规范

- 使用英文或中文保持团队一致。
- 每个提交应专注于一个逻辑改动，避免混杂无关修改。
- 不要提交未完成的临时改动。

---

### 二、版本发布与标签规范

版本号遵循 [语义化版本 2.0.0](https://semver.org/)：`主版本号.次版本号.修订号`

- **主版本**：不兼容的 API 变更。
- **次版本**：向下兼容的功能性新增。
- **修订号**：向下兼容的问题修复。

#### 1. 发布流程

1. 从 `develop`（或 `main`）创建 `release/x.y.z` 分支，进行最后的测试和文档更新。
2. 测试通过后，合并 `release` 分支到 `main` 和 `develop`。
3. 在 `main` 上打上对应版本的标签：`v1.2.3`。
4. 在 GitHub Releases 中创建发布说明，汇总主要变更。

#### 2. 标签命名

- 标签名以 `v` 开头，后跟版本号，如 `v1.0.0`、`v2.3.1-beta`。
- 轻量标签或附注标签均可，推荐使用附注标签（`-a`）记录作者、日期和说明。
